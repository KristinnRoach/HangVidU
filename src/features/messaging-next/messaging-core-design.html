Messaging Core Design

Minimal draft for a TS-first messaging module that supports direct and group chats, pluggable persistence, and private P2P delivery aligned with the current P2PRoom lifecycle.

Core owns message semantics Transport is swappable Group calls still WIP Private mode needs invitation flow

Proposed Layers

Solid UI Selected conversation, composer state, private toggle.
Messaging Service Open, subscribe, send, mark read, switch delivery mode.
Conversation Model IDs, participants, membership, message validation.
Delivery Router Chooses persistent repository or private P2P transport.
Backends RTDB repository, P2PRoom data channel, future backend.

Conversation Identity

Direct Chat

ID rule
{sortedUserA}_{sortedUserB}
Persistence path
conversations/{conversationId}
Compatibility
Current production format. Do not change without migration.

Group Chat

ID rule
group:{stableGeneratedId}
Membership
Stored as conversation metadata, not encoded in the ID.
Why
Members can be added/removed without changing history ID.

Decision Point

Recommended: keep deterministic IDs only for 1:1 direct chats using the current sorted underscore format. Use generated stable IDs for groups, because group membership changes over time and should not rewrite message history identity.

Backend Interface

Conversation Repository

  • loadConversation(id)
  • upsertConversation(record)
  • setDraft(id, draftOrNull)
  • subscribeConversation(id, callback)
  • Owns metadata, participants, delivery policy, and drafts.

Message Repository

  • loadMessages(id)
  • send(messageEnvelope)
  • subscribe(id, myUserId, callback)
  • setReaction(...)
  • subscribeReactions(id, callback)
  • Owns durable message envelopes and reaction deltas.

Private Transport

  • canDeliver(conversation)
  • connect(conversation)
  • send(protocolEnvelope)
  • subscribe(callback)
  • disconnect()
  • No durable storage contract.

Boundary Rule

Persistence is a repository concern. Private messaging is a delivery transport concern. Message creation should not know whether a message will be written to RTDB or sent over a data channel.

Delivery Policy

Persistent

  • Default mode.
  • Writes message envelopes to repository.
  • Works offline-ish when backend permits.
  • Supports unread and history.

Private In Active Room

  • Use current p2p.room().
  • Send to each remote member over room data channel.
  • Subscribe to dataChannelMessage.
  • Messages are local-session only.

Private Without Active Room

  • Create or request a private messaging room.
  • Join with dataChannel: true.
  • No media stream required if P2PRoom supports it.
  • Needs invitation/acceptance signaling.

P2PRoom Alignment

Current Shape To Match

  • p2p.join({ roomId, peerId, createSignaling, dataChannel: true })
  • p2p.room() returns active joined room.
  • room.send(memberId, data) sends one payload.
  • room.on('dataChannelMessage', handler) receives payloads.
  • room.dataChannels.get(memberId) checks channel readiness.

Messaging Protocol Over DataChannel

  • Use JSON string messages with protocol: 'hangvidu.messaging.v1'.
  • Ignore file-transfer payloads by protocol/type guard.
  • One envelope per message for text/events.
  • Large file payloads should stay in file-transfer protocol.
  • For group private mode, fan out one message per remote member.

Open Constraint

Current call join uses memberCapacity = 2. Group chat private mode should not depend on group-call readiness unless the P2PRoom library can support data-channel-only rooms with higher member capacity today.

Private Toggle Flow

User toggles Private UI asks Messaging Service to switch delivery policy.
Check room If active room covers participants, reuse its data channels.
Invite if needed Signal participants to join a private data-only room.
Confirm ready Private mode is enabled only when required channels are open.
Send private No repository write; local ephemeral history only.

Design Decisions To Settle

Group ID Strategy

Choose generated stable group: IDs, with membership in metadata. Avoid participant-list-derived group IDs.

Private Room Signaling

Decide whether private messaging room invites reuse call invite infrastructure or get their own lightweight conversation-session invite model.

Ephemeral History

Decide whether private messages remain visible only until page refresh, until conversation close, or until explicit clear.

Read State

Persistent messages support read state in the repository. Private messages can use transient delivery/read acknowledgements later.

First Implementation Slice

  • Added the canonical conversation repository boundary for metadata, participants, drafts, and delivery policy.
  • Added an in-memory conversation repository adapter for focused testing and early runtime wiring.
  • Keep repository and private transport boundaries centered on MessageEnvelope.
  • Treat the current RTDB adapter as a legacy-compatible adapter until the canonical persistence shape is decided.
  • Keep legacy messaging live until the new service is tested and can be wired one caller at a time.
